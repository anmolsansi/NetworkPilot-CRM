# TICKET R0.2: Restore calendar and extension route contracts

## Goal
Expose calendar and extension handlers at their documented `/api/v1` paths.

## Why This Exists
Feature routers and route decorators both add the resource prefix, producing undocumented double-prefixed 404s.

## User-Facing Behavior
Calendar link generation and all extension profile actions reach the backend again.

## Scope
- Change calendar decorator to `/daily-reminder-link` under the router prefix.
- Change extension decorators to `/lookup`, `/quick-create`, and `/quick-action` under the router prefix.

## Out of Scope
- Reminder delivery and favourite extension UI.
- Authentication changes.

## Files to Inspect First
- `backend/app/api/router.py`
- `backend/app/api/routes/calendar.py`
- `backend/app/api/routes/extension.py`

## Files to Create or Edit
- `backend/app/api/routes/calendar.py`
- `backend/app/api/routes/extension.py`
- `backend/app/tests/integration/test_extension_api.py`

## Data Model
None.

## API Contract
Authenticated routes remain `GET /api/v1/calendar/daily-reminder-link`, `GET /api/v1/extension/lookup`, `POST /api/v1/extension/quick-create`, and `POST /api/v1/extension/quick-action`; existing request/response schemas and error codes are unchanged.

## UI States
Existing callers retain their current states.

## Error Handling

## Auth & Permissions

Documented and frontend-consumed paths resolve exactly once.
