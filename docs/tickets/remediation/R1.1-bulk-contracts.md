# TICKET R1.1: Repair atomic bulk People contracts

## Goal
Make all seven bulk actions safe and compatible across v0.0.1, v0.0.5, and v0.0.9.

## Why This Exists
Merged tag/stage schema changes broke the existing client and CCR-001, and custom resource IDs are not workspace validated.

## User-Facing Behavior
Favourite, tag, priority, stage, archive, and next-action bulk controls complete atomically or show one safe error.

## Scope
- Implement CCR-006 request unions and normalization.
- Resolve tag names/IDs within the workspace.
- Validate custom stage ownership before any mutation.
- Preserve legacy stage/archive activity semantics.
- Eagerly return People relationships.

## Out of Scope
- Saved views and tag management screens.
- New bulk action kinds.

## Data Model
Existing `people`, `tags`, `person_tags`, `pipeline_stages`, and `activities`; no migration.

## API Contract
CCR-001 plus additive CCR-006. Existing auth/error/success shapes remain unchanged.

## UI States
Existing bulk bar loading, error, confirmation, and success behavior remains.

## Validation Rules
1–100 unique people; exactly one tag/stage selector family; 1–20 unique normalized tags/IDs; final 20 tags maximum; all resource IDs workspace-scoped.

## Error Handling
Validate every selected person/tag/stage before mutation; any failure rolls back the request.

## Auth & Permissions
Authenticated active workspace member, enforced by `require_workspace_access` and workspace predicates.

## Dependencies
R0.1 and CCR-006.

## Integration Contract
Both historical string payloads and identifier payloads remain supported.

## Implementation Steps
1. Expand strict payload validation.
2. Resolve resources before mutations.
3. Apply legacy/custom stage behavior and reload relationships.
4. Run old and new contract tests.

## Test Cases
Existing MAS-1 tests plus tag ID success/missing/cross-workspace and custom stage success/missing/cross-workspace.

## Manual QA
Select two People and run every action; confirm returned order and atomic errors.

## Acceptance Criteria
- [ ] All historical tests pass.
- [ ] Current frontend tag payload succeeds.
- [ ] Cross-workspace IDs cannot be attached.
- [ ] Activity history remains correct.

## Definition of Done
Apply `product_quality_bar.md`, `api_rules.md`, `security_rules.md`, and `testing_rules.md`.
